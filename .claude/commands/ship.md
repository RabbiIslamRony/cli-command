# Git Apply — Stage, Format, Commit, Branch & Push

Apply changes using the commit message, branch name, and PR details generated by `/git-gen`.

## Step 0: Load Git-Gen Output

Try to find the `/git-gen` output using this priority:

1. **Same session context** — If `/git-gen` was run in this conversation, use the output directly from context. Skip file read.
2. **Saved file** — Read `.claude/git-smart-output.json`. If it exists and has valid data, show the user what was found:
   - Branch name
   - Commit message
   - PR title (if any)
   - Timestamp of when it was generated

   Ask: "Ei git-gen output use korbo? (Generated: <timestamp>)"
   - Options: Yes, use this / No, run /git-gen first

3. **Nothing found** — If neither context nor file exists, ask the user using AskUserQuestion:

   "Git-gen output pawa jacche na. Ki korbo?"
   - Options:
     - Run /git-gen first (Recommended) — Stop and tell user to run `/git-gen`
     - I'll provide manually — Ask for commit message and branch name manually via AskUserQuestion

## Step 1: Stage All Changes

```bash
git add .
```

Show the user what was staged:

```bash
git status
```

## Step 2: Format Code (QA Checkpoint)

Ask the user using AskUserQuestion:

"yarn format & composer format run korbo?"

- Options:
  - Run both (Recommended)
  - Run yarn format only
  - Run composer format only
  - Skip formatting

Based on user selection:

- **yarn format**: Run `yarn format`. If it fails, show the error and ask the user how to proceed.
- **composer format**: Run `composer format`. If it fails, show the error and ask the user how to proceed.
- **Skip**: Move to Step 3.

## Step 3: Re-stage After Formatting

Run `git status` to check if formatting created new changes.

- If there are new unstaged changes: run `git add .` and notify the user.
- If there are unexpected issues or errors: ask the user before proceeding.
- If clean: move to Step 4.

## Step 4: Commit

Use the commit message from git-smart output (loaded in Step 0).

```bash
git commit -m "<commit message from git-smart>"
```

If commit fails (e.g., pre-commit hook), show the error and ask the user how to proceed. Do NOT amend — create a new commit after fixing.

## Step 5: Create Branch

Create a new branch using the branch name from git-smart output (loaded in Step 0):

```bash
git checkout -b <branch-name-from-git-smart>
```

If the branch already exists, ask the user:

"Ei branch ta already ache. Ki korbo?"

- Options:
  - Switch to it (checkout existing)
  - Pick a different name (ask for new name)
  - Force recreate (delete and recreate)

## Step 6: Push to Remote (QA Checkpoint)

First, list available remotes and current remote branches:

```bash
git remote -v
git branch -r --sort=-committerdate | head -20
```

Ask the user using AskUserQuestion:

"Kon remote-e push korbo?"

- Show detected remotes as options (e.g., `origin`, `upstream`, etc.)
- Include options:
  - Push to origin/<current-branch> (default)
  - Push to a different remote (ask which one)
  - Don't push now

If user selects a remote:

```bash
git push -u <remote> <branch-name>
```

If push fails (e.g., branch exists on remote), show error and ask whether to force push or pick a different branch name.

## Step 7: Create PR (Optional)

If git-smart output has a PR title and description, ask:

"PR create korbo?"

- Options:
  - Yes, create PR now
  - No, skip PR

If yes:

```bash
gh pr create --title "<PR title from git-smart>" --body "<PR description from git-smart>"
```

Return the PR URL to the user.

## Step 8: Cleanup

After successful completion (commit + push done), delete `.claude/git-smart-output.json` so stale data doesn't confuse future runs.

## Important Rules

- ALWAYS try to load git-smart output first (Step 0) — never skip
- NEVER skip QA checkpoints — always ask the user at format and push steps
- NEVER force push without explicit user approval
- NEVER stage files that contain secrets (.env, credentials, API keys)
- If any step fails, show the error and ask — do NOT silently retry or skip
- Always show `git status` after staging so the user can verify
- Delete the output file after successful apply to keep things clean
