# Project Context Extractor

Extract and save full project context to `CLAUDE.md` at the project root. Claude Code automatically loads `CLAUDE.md` in every conversation, so all commands get project context without needing to read a separate file.

## Step 0: Check Existing Context

First check if `CLAUDE.md` already exists at the project root.

- If it exists, read it and ask using AskUserQuestion:
  "CLAUDE.md already ache. Ki korbo?"
  - Options:
    - Update it (re-scan + keep manual notes)
    - Overwrite completely (fresh start)
    - Cancel — current context is fine

- If it does NOT exist, proceed to Step 1.

## Step 1: Auto-Scan Codebase

Launch these agents IN PARALLEL to extract project info automatically:

### Agent 1: Tech Stack Detector
Scan for:
- `composer.json` — PHP dependencies, autoload, project name
- `package.json` — JS/Node dependencies, scripts, build tools
- `webpack.config.js` / `vite.config.js` / `gulpfile.js` — build system
- `phpcs.xml` / `.eslintrc` / `.prettierrc` — coding standards
- `.editorconfig` — editor settings
- WordPress plugin header (main plugin file) — plugin name, version, description, author, requires
- `readme.txt` — WordPress.org description, FAQ, changelog
- Report: full tech stack with versions

### Agent 2: Project Structure Analyzer
Scan for:
- Top-level directory structure (`ls` the root)
- Key directories and their purpose (assets/, includes/, templates/, views/, etc.)
- Entry points (main plugin file, REST API endpoints, AJAX handlers)
- Template hierarchy (if WordPress theme/plugin)
- Asset pipeline (SCSS source → compiled CSS, JS source → bundled JS)
- Database usage (custom tables, options, meta)
- Report: directory map with purpose of each major folder

### Agent 3: Workflow Detector
Scan for:
- `.github/workflows/` — CI/CD pipelines
- Git branching strategy — check branch names: `git branch -r | head -30`
- Main/default branch name
- Release pattern — check tags: `git tag --sort=-version:refname | head -10`
- Recent commit message style: `git log --oneline -15`
- Build commands from `package.json` scripts
- Coding conventions — check a few PHP and JS files for patterns (naming, spacing, hooks usage)
- Report: workflow summary

## Step 2: Ask User for Non-Detectable Info

After auto-scan completes, ask using AskUserQuestion (single call, multiple questions):

1. **Project Purpose** — "Ei project ta ki kore? One-line description dao."
   - Options:
     - WordPress plugin (I'll describe)
     - WordPress theme
     - Full-stack application
     - Library/package
   - multiSelect: false

2. **Target Audience** — "Ei project kader jonno?"
   - Options:
     - End users (non-technical)
     - Developers
     - Site administrators
     - Mix — both users and developers
   - multiSelect: false

3. **Team Size** — "Ei project e kojon kaj kore?"
   - Options:
     - Solo developer (just me)
     - Small team (2-5)
     - Medium team (6-15)
     - Large team (15+)
   - multiSelect: false

4. **Deploy Process** — "Deploy kivabe hoy?"
   - Options:
     - WordPress.org SVN
     - GitHub release → auto deploy
     - Manual upload
     - CI/CD pipeline
   - multiSelect: false

## Step 3: Generate & Save Context

Combine auto-scan results + user answers into `CLAUDE.md` at the project root:

```markdown

> Auto-generated by `/context` command — Last updated: YYYY-MM-DD
> Run `/context` again to update

## Project

- **Name:** <from plugin header or package.json>
- **Description:** <one-line from user or readme>
- **Type:** <wordpress-plugin | wordpress-theme | app | library>
- **Version:** <current version>
- **Author:** <from plugin header>
- **License:** <from composer/package>

## Audience

- **Target:** <end-users | developers | admins | mix>
- **Team Size:** <solo | small | medium | large>

## Tech Stack

- **Language:** <PHP 7.4+ | PHP 8.0+ etc>
- **Framework:** <WordPress 5.x+ | Laravel | etc>
- **CSS:** <SCSS | Tailwind | plain CSS>
- **JS:** <Vanilla | jQuery | React | Vue>
- **Build:** <Webpack | Vite | Gulp | none>
- **Package Manager:** <npm | yarn | pnpm>
- **PHP Deps:** <Composer | none>

## Structure

- **Entry Point:** <main plugin file>
- **Assets Source:** <assets/src/>
- **Assets Compiled:** <assets/css/, assets/js/>
- **Templates:** <templates/ | views/>
- **Includes:** <includes/>
- **REST API:** <path if exists>
- **AJAX Handlers:** <path if exists>
- **Database:**
  - Custom Tables: <yes/no — list if yes>
  - Options Prefix: <prefix>
  - Meta Usage: <post_meta, user_meta, etc>

## Workflow

- **Main Branch:** <trunk | main | master>
- **Branch Strategy:** <feature branches | gitflow | trunk-based>
- **Commit Style:** <conventional | freeform — example>
- **Build Command:** <npm run build | gulp | etc>
- **Watch Command:** <npm run watch | etc>
- **Deploy:** <wordpress.org | github-release | manual | ci-cd>
- **CI/CD:** <github-actions | none | other>

## Conventions

- **PHP Standard:** <WordPress | PSR-12 | custom>
- **JS Standard:** <eslint config | none>
- **Naming:**
  - PHP Functions: <snake_case>
  - PHP Classes: <PascalCase>
  - JS Functions: <camelCase>
  - CSS Classes: <BEM | directorist- prefix | etc>
- **Hooks Prefix:** <directorist_ | atbdp_ | etc>
- **Text Domain:** <directorist | etc>

## Notes

<any special notes from user or detected quirks>
```

After saving, show the user:

```

Summary:
- Project: [name] — [description]
- Stack: [PHP x.x + WordPress + SCSS + JS]
- Build: [build tool + command]

Claude Code ekhon automatically ei context load korbe prottek conversation-e.
Kono command ke alada kore context read korte hobe na.
/context abar run korle update korte parbe.
```

## Important Rules

- NEVER overwrite existing CLAUDE.md without asking first
- ALWAYS auto-detect as much as possible — minimize user questions
- ALWAYS include `Last updated` date in the header
- Keep markdown clean and readable — someone should be able to read it manually
- If a field can't be detected and user doesn't answer, use "unknown" — don't guess
- The CLAUDE.md file should be committed to git (it's project info, not secrets)
- Maximum 2 rounds of user questions — 1 main + 1 follow-up if needed
- If updating, preserve user's manual edits/notes
- If an old `.claude/context.yaml` exists, migrate its data into the new `CLAUDE.md` and inform the user they can delete the old file
